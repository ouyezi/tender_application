## 1. 目标

结合以下输入，通过可插拔智能体为每个任务生成独立的诊断检查项清单：

1. 招标文件解析正文
2. 完整招标文件解读报告
3. 管理端诊断配置（软性参考）

每条检查项必须包含诊断标题、诊断要求、诊断技巧、重要性和动态分类，并提供可供后续工作区召回与自动判定使用的机器字段。描述应足够精确，使后续诊断能够找到相关标书内容，并区分符合性结论与失分、废标等后果。

任务详情报告区调整为三个 Tab：

`解读报告 → 检查项报告 → 诊断报告`

### 1.1 成功标准

1. 每个任务都能基于招标正文、解读报告和管理配置生成任务级检查项。
2. 检查项具有完整展示字段、来源依据、检索提示、预期证据和结构化判定规则。
3. 智能体按预计命中相似标书内容块的原则动态分类。
4. 短文档单次生成；长文档分片提取，并通过稳定提示词前缀利用模型缓存。
5. 检查项报告位于解读报告之后、诊断报告之前。
6. 本期 Mock 诊断只消费生成的任务级检查项，并按分类批量执行。
7. 解析或生成失败时不静默降级，提供明确失败阶段和重试路径。

### 1.2 本期不做

- 不接入真实 HTTP 智能体或模型 SDK。
- 不实现真实工作区文档召回；只定义 `RetrievalProvider` 接口并使用 Mock。
- 不实现检查项报告下载。
- 不保留智能体返回但未通过校验的半成品正式清单。
- 不使用固定业务分类枚举。

## 2. 总体架构与流水线

任务创建后，招标文件解析与现有解读流程并行执行。解读成功后进入检查项生成阶段；该阶段必须等待招标正文解析成功。

```text
创建任务
  ├─ 招标文件工作区解析
  └─ interpreting：生成解读报告
           │
           ▼
generating_checklist：等待正文 → 生成并校验检查项
           │
           ▼
diagnosing：按动态分类批量执行 Mock 诊断
           │
           ▼
completed：生成诊断报告
```

主状态机：

```text
interpreting → generating_checklist → diagnosing → completed
      │                  │                 │
      └──────────────────┴─────────────────┴─→ failed / stopped
                                          
diagnosing ↔ paused
```

- `pause` / `resume` 仍只允许用于 `diagnosing` / `paused`。
- `stop` 允许用于所有非终态。
- 进程重启时，`interpreting`、`generating_checklist`、`diagnosing`、`paused` 和旧状态 `running` 均恢复为 `stopped`。
- 检查项生成成功后才进入诊断；诊断开始后清单冻结。
- 招标正文解析失败时，任务进入 `failed`，不允许仅靠解读报告或管理配置降级生成。
- 解读已成功时，正文重新解析和检查项重新生成不重复执行解读。

## 3. 模块边界

### 3.1 `ChecklistContextBuilder`

负责准备智能体输入：

- 读取带章节、页码或段落定位的招标正文。
- 读取完整解读报告。
- 序列化管理端配置快照。
- 估算 token，选择单次或分片生成路径。
- 生成稳定提示词前缀和正文分片。

它不负责调用智能体、校验输出或写数据库。

### 3.2 `ChecklistAgent`

可插拔检查项生成协议。本期提供确定性 `MockChecklistAgent`，未来 HTTP 实现遵循同一输入输出 Schema。

职责：

- 从完整正文或正文分片提取候选检查项。
- 全局合并、去重并补全判定信息。
- 根据预计命中的标书内容位置动态分类。

### 3.3 `ChecklistService`

负责：

- 调用 `ChecklistAgent`。
- 校验智能体结构化输出。
- 原子持久化生成记录、分类和检查项。
- 同步结构化 JSON 到任务 Artifact。
- 管理检查项生成重试。

### 3.4 `BatchDiagnosisEngine`

以分类为执行单位：

```text
diagnose_category(category, items, retrieved_chunks) -> item_results
```

返回结果必须通过检查项 ID 与输入完整映射。漏项、重复项或未知 ID 均视为整个批次失败，不写入部分正式结果。

### 3.5 `RetrievalProvider`

为分类诊断返回多个合并去重后的标书内容块。本期返回 Mock 内容；后续工作区检索功能替换实现，不改变调度器和批量诊断协议。

### 3.6 `Scheduler`

只负责编排阶段、状态转换和停止/暂停控制，不承载上下文组装、智能体输出校验或业务规则。

## 4. 数据模型

### 4.1 `checklist_generations`

每次生成尝试对应一条记录：

- `id`
- `task_id`
- `status`：`generating` | `succeeded` | `failed`
- `agent_type`
- `agent_version`
- `schema_version`
- `input_hash`
- `admin_config_snapshot`
- `raw_response_path`：保存原始响应用于排查，不作为正式清单
- `error_message`
- `created_at` / `finished_at`

`diagnosis_tasks` 新增 `current_checklist_generation_id`。任务只有一个当前生效的成功版本；生成期间该字段保持原值或为空，只有新版本完整校验并原子写入成功后才切换。失败重试创建新的生成记录，历史失败记录保留。

### 4.2 `checklist_categories`

动态分类及后续批诊断边界：

- `id`
- `generation_id`
- `name`
- `description`
- `retrieval_query`
- `expected_locations`
- `sort_order`

分类名称由智能体根据招标文件动态生成，不受固定业务枚举限制。分类原则是其中检查项预计会命中相似的标书内容块，而不是仅按抽象业务主题归类。

每个分类规模必须处于配置上限内。分类过大时，生成阶段按预计标书位置拆分为更具体的分类，以维持“一分类一次诊断调用”。

### 4.3 `checklist_items`

每条检查项只属于一个主分类：

- `id`
- `generation_id`
- `category_id`
- `title`：诊断标题
- `requirement`：诊断要求
- `technique`：诊断技巧
- `importance`：`high` | `medium` | `low`
- `source_references`：招标依据及章节、页码或段落定位
- `retrieval_hints`：补充分类级检索的关键词、同义表达和关联位置
- `expected_evidence`：判定所需的内容或材料
- `compliance_rules`：符合性判定规则
- `consequence_rules`：后果判定规则
- `admin_config_refs`：相关管理配置 ID，可为空
- `sort_order`
- `created_at`

管理配置是软性参考。智能体可以采用、合并或忽略配置内容，不要求每条管理配置在结果中都有映射。

### 4.4 诊断结果

现有诊断结果模型需关联 `checklist_item_id`，并支持两维结果：

1. 符合性：`satisfied`（满足）、`violated`（违反）、`cannot_satisfy`（不能满足）、`insufficient_evidence`（证据不足）。
2. 后果标签：`no_score`（无法得分）、`bid_unusable`（废标/标书不可用）、`score_risk`（扣分风险）、`general_risk`（一般风险），可多选。

证据、说明和建议继续作为结果字段保留。

## 5. 检查项内容标准

每条检查项只描述一个可独立判断的要点。

### 5.1 诊断要求

必须明确：

- 标书中必须出现的内容或材料。
- 允许的形式、范围、数值、时间或资格条件。
- 禁止出现的偏差或冲突。
- 缺失、矛盾或不完整时应如何判断。
- 与其他章节或材料需要交叉核对的关系。

### 5.2 诊断技巧

必须明确：

- 优先检索的标书章节或材料类型。
- 关键词、同义表达和可能的替代表述。
- 需要召回的多个相关内容块。
- 内容块合并后应进行的交叉核对。
- 容易出现误判的边界情况。

### 5.3 来源与判定

- 每条检查项必须包含可追溯的招标依据。
- 无招标依据的推断项不得生成。
- 符合性规则和后果规则分别表达，避免将“违反”和“无法得分”等非互斥概念混为单一状态。
- 高度重叠的要求合并；存在不同判定条件的要求不得因文字相似而错误合并。

## 6. 长短文档生成策略

### 6.1 短文档

先估算招标正文 token 数。正文不超过可配置阈值时，一次调用生成完整候选清单，再执行服务端校验和持久化。

### 6.2 长文档

超过阈值时：

1. 优先按章节切分。
2. 单章节仍过长时再按 token 切分。
3. 相邻分片保留可配置重叠，并保留章节、页码或段落定位。
4. 各分片提取带来源引用的候选检查项。
5. 全部分片完成后执行一次全局合并、去重、规则补全和动态分类。

### 6.3 缓存友好提示词

每次分片调用使用相同的稳定前缀，顺序固定：

1. 固定系统指令和输出 Schema
2. 完整解读报告
3. 管理端配置快照
4. 当前招标正文分片

只有最后的正文分片变化，以最大化模型提示词前缀缓存命中。

以下参数配置化：

- 单次生成 token 阈值
- 分片 token 大小
- 分片重叠大小
- 单分类最大检查项数

## 7. 输出校验与原子写入

服务端不直接信任智能体输出。正式写入前必须校验：

- Schema 版本和必填字段。
- 重要性与判定规则枚举。
- ID 唯一性。
- 分类存在且归属唯一。
- 来源引用非空且指向有效正文范围。
- 每个分类具有共享检索策略。
- 标题、要求和技巧不为空且不重复。
- 每条输入检查项在批量诊断结果中有且只有一个输出。

校验失败时：

- 不写入半成品正式分类或检查项。
- 保存原始响应和校验错误。
- 将任务标记为 `failed`，记录失败阶段。
- 允许用户在问题修复后只重试检查项生成。

## 8. API

### 8.1 检查项查询

`GET /api/tasks/{task_id}/checklist`

返回：

- 当前生成记录及状态。
- 分类列表。
- 每个分类下的检查项。
- 汇总数量。

检查项报告独立按需加载，不扩充现有 `TaskOut` 的主体响应。

### 8.2 重试

`POST /api/tasks/{task_id}/checklist/retry`

- 只允许在检查项生成失败且招标正文解析已成功时调用。
- 若招标正文解析失败，返回 `409` 并要求先使用现有工作区重新解析接口。
- 解读报告已存在时不重新解读。
- 诊断已开始后不允许重生成清单。

### 8.3 错误语义

- 任务或生成记录不存在：`404`
- 当前状态不允许操作：`409`
- 智能体结构化输出不合法：任务进入 `failed`，详情返回明确失败阶段和原因

## 9. 前端设计

任务详情报告区的 Tab 顺序：

1. 解读报告
2. 检查项报告
3. 诊断报告

### 9.1 检查项报告

顶部展示：

- 生成状态
- 检查项总数
- 分类数
- 各重要性数量

主体展示：

- 按动态分类筛选。
- 表格列为诊断标题、诊断要求、诊断技巧、重要性、分类。
- 每行可展开查看招标依据、检索提示、预期证据、符合性规则和后果规则。

### 9.2 页面状态

- `generating_checklist`：展示等待正文或正在生成的状态。
- 生成成功：展示冻结的检查项清单。
- 解析失败：展示解析失败原因和前往工作区重新解析的入口。
- 生成失败：展示失败阶段、原因和重试入口。
- 旧任务无检查项：展示兼容性空态，不伪造报告。

本期不增加检查项报告下载。结构化 JSON 同步到任务 Artifact，界面通过 API 展示。

## 10. 本期 Mock 诊断与未来召回

生成成功后，诊断进度总数来自任务级检查项，不再来自创建任务时的全局配置快照。

本期流程：

1. 按动态分类加载检查项。
2. `RetrievalProvider` 返回 Mock 内容块。
3. `MockBatchDiagnosisEngine` 每个分类调用一次。
4. 按检查项 ID 返回确定性的符合性、后果标签、证据、说明和建议。
5. 全部分类完成后生成现有诊断报告。

未来工作区召回：

1. 使用分类级 `retrieval_query` 和 `expected_locations` 发起主召回。
2. 合并分类内各检查项的 `retrieval_hints` 补充召回。
3. 召回多个内容块，去重并控制 token 预算。
4. 将合并内容块和分类下全部检查项交给真实 `BatchDiagnosisEngine`。

未来替换不改变检查项数据模型、调度流程和前端报告结构。

## 11. 错误处理与恢复

- 招标正文解析失败：任务失败，不降级生成；先重新解析，再重试检查项生成。
- 解读失败：沿用现有行为，不进入检查项生成。
- 智能体调用失败：保存生成尝试和错误，可重试。
- 输出校验失败：保存原始响应和校验详情，不写正式清单。
- 分类批诊断失败：整批不写部分结果，任务失败并保留已完成分类结果。
- 进程重启：非终态任务标为 `stopped`，不自动继续内存任务。
- 管理配置后续修改：不影响已生成并冻结的任务级检查项。

## 12. 测试与验收

### 12.1 自动化测试

至少覆盖：

1. 短正文走单次生成。
2. 长正文按章节/token 分片。
3. 分片调用固定前缀完全一致，仅正文分片变化。
4. 全局合并后无重复，来源引用可追溯。
5. 缺字段、非法重要性、未知分类、重复 ID 或无来源依据时校验失败。
6. 动态分类、生成记录和检查项原子持久化。
7. 管理配置仅作软参考，不强制逐项映射。
8. 正文解析失败阻止生成；重新解析和重试可恢复且不重复解读。
9. 诊断进度总数来自生成检查项。
10. Mock 按分类批量返回全部逐项结果。
11. 漏项、重复项或未知检查项 ID 导致批次失败。
12. 诊断开始后清单不可变。
13. 三个 Tab 的加载、成功、失败和重试状态。
14. 数据库迁移、旧任务兼容和进程重启恢复。

### 12.2 手动验收

1. 创建任务并看到状态依次进入解读、检查项生成、诊断和完成。
2. 检查项 Tab 位于解读报告与诊断报告之间。
3. 每条检查项展示五个主要字段，并可展开查看机器字段。
4. 分类名称由招标内容动态产生，同类检查项预计命中相似标书位置。
5. 招标解析失败时无不完整清单，并可通过重新解析和重试恢复。
6. 管理配置修改不影响已生成任务。
7. Mock 诊断结果与生成检查项逐项对应。

## 13. 兼容与迁移

- 新增表使用现有轻量 SQLite 迁移机制创建。
- 旧任务保留原 `config_snapshot` 和诊断结果，不强制补生成检查项。
- 新任务的 `config_snapshot` 可继续保存创建时管理配置快照用于审计，但不作为诊断执行清单。
- 前端对无检查项的旧任务展示兼容性空态；原解读报告和诊断报告仍可访问。
- 原 `running` 状态继续作为历史兼容值处理，新任务不再写入。

## 14. 决议摘要

- 使用任务级生成记录、动态分类和检查项三层模型。
- 生成输入为招标正文、完整解读报告和管理配置软参考。
- 分类由智能体动态生成，并作为未来召回和批诊断边界。
- 短文档单次生成；长文档使用稳定前缀分片提取并全局合并。
- 判定拆为符合性与后果标签两维。
- 解析失败不降级，必须重新解析后重试。
- 本期实现协议、确定性 Mock、结构化校验和分类批量 Mock 诊断。
- 任务详情使用“解读报告 → 检查项报告 → 诊断报告”三个 Tab。
